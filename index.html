03 — Selected Work

Case studies

Oracy Scoring AI

Solo · end-to-end

Context

Language programs needed a way to assess spoken English at scale. Manual listening and scoring per student didn't scale across cohorts, and recordings lived in Zoom Cloud with no structured pipeline to get from "audio exists" to "a teacher can see a score."

Constraints

Built and run solo, so constraints were as much about minimizing ongoing operational burden as solving the assessment problem itself: a rate-limited speech API, recordings long enough (60+ minutes) to overflow that API's buffer, and no appetite for exposing a public endpoint just to receive webhooks.

Approach

Designed an event-driven pipeline that polls for new recordings, normalizes per-student audio, then runs a hybrid scoring model — pronunciation and fluency from a speech-specific assessment service, vocabulary/grammar/topic relevance from an LLM reading the transcript.

  • Polling over webhooksAvoided requiring a public endpoint, simplifying deployment, while leaving room to add webhooks later without touching the core pipeline.
  • Human-in-the-loop name resolutionFuzzy matching with a confidence threshold, falling back to manual review for anything ambiguous. Scoring waits until every recording in a session is resolved.
  • Two isolated worker queuesOne for I/O-bound work, one dedicated to the speech-scoring API, sized to respect that provider's rate limits.
  • Composite scoring logicA weighted formula that penalizes weak components more than it rewards strong ones, so one weak skill can't be masked by a strong one.

Outcome

Across a year of class recordings, the pipeline identified each student's key improvement areas and surfaced their single most prominent pronunciation error — giving teachers a clear, individualized focus point instead of an unstructured transcript to review by hand.

AI-Assisted Incident Response for Data Pipelines

Workflow design & ownership

Context

At a company running enterprise data-ingestion pipelines for multiple customers, failures were detected automatically, but everything after detection — investigation, context-gathering, ownership, tracking — stayed manual. That gap was the real bottleneck, not detection itself.

Constraints

The agent needed real operational context — logs, customer metadata, prior incidents — to investigate well, but that context sits next to sensitive customer data across many accounts. Tool access had to be tightly scoped, and impossible to accidentally widen as new failure types got added later.

Approach

Designed a workflow where an agent — given structured access to pipeline logs, customer metadata, and prior-incident history — investigates each failure, proposes a remediation, and opens a pull request once its confidence in that fix clears a set threshold. Each failure type is handled by its own written playbook, automatically routed by issue label, with the data it can read declared per flow.

Simplified — amber: agent-driven, teal: human-reviewed

  • Confidence-based branchingRather than treating every failure the same, the workflow uses a confidence threshold to decide the level of autonomy: high-confidence fixes become pull requests, lower-confidence investigations become structured handover reports — because an incorrect autonomous fix is more expensive than a slower, well-informed human one.
  • Human review as the gateEvery path ends with human review before anything ships. The agent proposes, it doesn't deploy — that boundary is what keeps engineering accountability intact as autonomy increases.
  • PRs, not direct deploymentRouting fixes through a pull request — rather than auto-deploying — keeps the agent inside the same review, CI, and rollback safeguards every other code change goes through.
  • Feedback loopOutcomes from review — accepted as-is, corrected, or rejected — feed a metrics loop used to keep improving the agent over time, so even a wrong investigation still adds value to the next one.
  • Scoped data, centrally-locked permissionsWhat data a flow can read is configurable per failure type, but what actions the agent is allowed to take is locked at the automation level, separate from that config — so adding a new failure type can change what the agent sees, but never what it's allowed to do.
