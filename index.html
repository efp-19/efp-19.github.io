02 — Selected Work

Case studies

Oracy Scoring AI

Solo · end-to-end

Context

Language programs needed a way to assess spoken English at scale. Manual listening and scoring per student didn't scale across cohorts, and recordings lived in Zoom Cloud with no structured pipeline to get from "audio exists" to "a teacher can see a score."

Approach

Designed an event-driven pipeline that polls for new recordings, normalizes per-student audio, then runs a hybrid scoring model — pronunciation and fluency from a speech-specific assessment service, vocabulary/grammar/topic relevance from an LLM reading the transcript.

  • Polling over webhooksAvoided requiring a public endpoint, simplifying deployment, while leaving room to add webhooks later without touching the core pipeline.
  • Human-in-the-loop name resolutionFuzzy matching with a confidence threshold, falling back to manual review for anything ambiguous. Scoring waits until every recording in a session is resolved.
  • Two isolated worker queuesOne for I/O-bound work, one dedicated to the speech-scoring API, sized to respect that provider's rate limits.
  • Composite scoring logicA weighted formula that penalizes weak components more than it rewards strong ones, so one weak skill can't be masked by a strong one.

Outcome

Across a year of class recordings, the pipeline identified each student's key improvement areas and surfaced their single most prominent pronunciation error — giving teachers a clear, individualized focus point instead of an unstructured transcript to review by hand.

AI-Assisted Incident Response for Data Pipelines

Lead Solution Engineer

Context

At a company running enterprise data-ingestion pipelines for multiple customers, failures were detected automatically, but everything after detection — investigation, context-gathering, ownership, tracking — stayed manual. That gap was the real bottleneck, not detection itself.

Approach

Designed a workflow where an agent, given access to a defined set of internal tools — logs, customer metadata, prior-incident history — investigates each failure and attempts a fix on its own.

  • Confidence-based branchingIf the agent is confident in its fix, it opens a pull request directly. If not, it produces a structured handover document instead, so an engineer isn't starting from zero.
  • Human review as the gateEither path ends with human review before anything ships — the agent proposes, it doesn't deploy.
  • Feedback loopOutcomes from review (PR accepted as-is, fix was wrong, handover was useful or wasn't) feed a metrics loop used to keep improving the agent over time.
